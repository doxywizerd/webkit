48  // This offset represents the constant register offset to the stored symbol table that represents the layout of the
49  // module environment. This symbol table is created by the byte code generator since the module environment includes
50  // the top-most lexical captured variables inside the module code. This means that, once the module environment is
51  // allocated and instantiated from this symbol table, it is titely coupled with the specific unlinked module program
52  // code block and the stored symbol table. So before executing the module code, we should not clear the unlinked module
53  // program code block in the module executable. This requirement is met because the garbage collector only clears
54  // unlinked code in (1) unmarked executables and (2) function executables.
55  //
56  // Since the function code may be executed repeatedly and the environment of each function execution is different,
57  // the function code need to allocate and instantiate the environment in the prologue of the function code. On the
58  // other hand, the module code is executed only once. So we can instantiate the module environment outside the module
59  // code. At that time, we construct the module environment by using the symbol table that is held by the module executable.
60  // The symbol table held by the executable is the cloned one from one in the unlinked code block. Instantiating the module
61  // environment before executing and linking the module code is required to link the imported bindings between the modules.
62  //
63  // The unlinked module program code block only holds the pre-cloned symbol table in its constant register pool. It does
64  // not hold the instantiated module environment. So while the module environment requires the specific unlinked module
65  // program code block, the unlinked module code block can be used for the module environment instantiated from this
66  // unlinked code block. There is 1:N relation between the unlinked module code block and the module environments. So the
67  // unlinked module program code block can be cached.
68  //
69  // On the other hand, the linked code block for the module environment includes the resolved references to the imported
70  // bindings. The imported binding references the other module environment, so the linked code block is titly coupled
71  // with the specific set of the module environments. Thus, the linked code block should not be cached.
72  int moduleEnvironmentSymbolTableConstantRegisterOffset() { return m_moduleEnvironmentSymbolTableConstantRegisterOffset; }

74  {
75  m_moduleEnvironmentSymbolTableConstantRegisterOffset = offset;
76  }
