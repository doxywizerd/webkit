108  // Ideally, there would only be one allocation profile for subclassing but due to Reflect.construct we
109  // have two. There are some pros and cons in comparison to our current system to using the same profile
110  // for both JS constructors and subclasses of builtin constructors:
111  //
112  // 1) + Uses less memory.
113  // 2) + Conceptually simplier as there is only one profile.
114  // 3) - We would need a check in all JSFunction object creations (both with classes and without) that the
115  // new.target's profiled structure has a JSFinalObject ClassInfo. This is needed, for example, if we have
116  // `Reflect.construct(Array, args, myConstructor)` since myConstructor will be the new.target of Array
117  // the Array constructor will set the allocation profile of myConstructor to hold an Array structure
118  //
119  // We don't really care about 1) since this memory is rare and small in total. 2) is unfortunate but is
120  // probably outweighed by the cost of 3).
121  ObjectAllocationProfile m_objectAllocationProfile;
122  InlineWatchpointSet m_objectAllocationProfileWatchpoint;
123  InternalFunctionAllocationProfile m_internalFunctionAllocationProfile;
124  WriteBarrier<Structure> m_boundFunctionStructure;
125  bool m_hasReifiedLength { false };
126  bool m_hasReifiedName { false };
