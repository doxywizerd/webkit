37 // An AccessCase describes one of the cases of a PolymorphicAccess. A PolymorphicAccess represents a
38 // planned (to generate in future) or generated stub for some inline cache. That stub contains fast
39 // path code for some finite number of fast cases, each described by an AccessCase object.
40 //
41 // An AccessCase object has a lifecycle that proceeds through several states. Note that the states
42 // of AccessCase have a lot to do with the global effect epoch (we'll say epoch for short). This is
43 // a simple way of reasoning about the state of the system outside this AccessCase. Any observable
44 // effect - like storing to a property, changing an object's structure, etc. - increments the epoch.
45 // The states are:
46 //
47 // Primordial: This is an AccessCase that was just allocated. It does not correspond to any actual
48 // code and it is not owned by any PolymorphicAccess. In this state, the AccessCase
49 // assumes that it is in the same epoch as when it was created. This is important
50 // because it may make claims about itself ("I represent a valid case so long as you
51 // register a watchpoint on this set") that could be contradicted by some outside
52 // effects (like firing and deleting the watchpoint set in question). This is also the
53 // state that an AccessCase is in when it is cloned (AccessCase::clone()).
54 //
55 // Committed: This happens as soon as some PolymorphicAccess takes ownership of this AccessCase.
56 // In this state, the AccessCase no longer assumes anything about the epoch. To
57 // accomplish this, PolymorphicAccess calls AccessCase::commit(). This must be done
58 // during the same epoch when the AccessCase was created, either by the client or by
59 // clone(). When created by the client, committing during the same epoch works because
60 // we can be sure that whatever watchpoint sets they spoke of are still valid. When
61 // created by clone(), we can be sure that the set is still valid because the original
62 // of the clone still has watchpoints on it.
63 //
64 // Generated: This is the state when the PolymorphicAccess generates code for this case by
65 // calling AccessCase::generate() or AccessCase::generateWithGuard(). At this point
66 // the case object will have some extra stuff in it, like possibly the CallLinkInfo
67 // object associated with the inline cache.
68 // FIXME: Moving into the Generated state should not mutate the AccessCase object or
69 // put more stuff into it. If we fix this, then we can get rid of AccessCase::clone().
70 // https://bugs.webkit.org/show_bug.cgi?id=156456
71 //
72 // An AccessCase may be destroyed while in any of these states.
73 //
74 // We will sometimes buffer committed AccessCases in the PolymorphicAccess object before generating
75 // code. This allows us to only regenerate once we've accumulated (hopefully) more than one new
76 // AccessCase.
