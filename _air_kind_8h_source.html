36 // Air opcodes are always carried around with some flags. These flags are understood as having no
37 // meaning if they are set for an opcode to which they do not apply. This makes sense, since Air
38 // is a complex instruction set and most of these flags can apply to basically any opcode. In
39 // fact, it's recommended to only represent something as a flag if you believe that it is largely
40 // opcode-agnostic.

79  // This is an opcode-agnostic flag that indicates that we expect that this instruction will
80  // trap. This causes the compiler to assume that this side-exits and therefore has non-control
81  // non-arg effects. This also causes the compiler to tell you about all of these instructions.
82  // Note that this is just one of several ways of supporting trapping in Air, and it's the less
83  // precise variant because it's origin-based. This means that if an instruction was fused out
84  // of B3 values that had different origins, then the origin at which you'll appear to trap
85  // will be somewhat random. The upside of this approach is that it imposes by far the least
86  // overhead on the compiler.
