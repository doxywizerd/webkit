60  // This gets called on for each Inst that uses this Special. Note that there is no way to
61  // guarantee that a Special gets used from just one Inst, because Air might taildup late. So,
62  // if you want to pass this information down to generate(), then you have to either:
63  //
64  // 1) Generate Air that starts with a separate Special per Patch Inst, and then merge
65  // usedRegister sets. This is probably not great, but it optimizes for the common case that
66  // Air didn't duplicate code or that such duplication didn't cause any interesting changes to
67  // register assignment.
68  //
69  // 2) Have the Special maintain a HashMap<Inst*, RegisterSet>. This works because the analysis
70  // that feeds into this call is performed just before code generation and there is no way
71  // for the Vector<>'s that contain the Insts to be reallocated. This allows generate() to
72  // consult the HashMap.
73  //
74  // 3) Hybrid: you could use (1) and fire up a HashMap if you see multiple calls.
75  //
76  // Note that it's not possible to rely on reportUsedRegisters() being called in the same order
77  // as generate(). If we could rely on that, then we could just have each Special instance
78  // maintain a Vector of RegisterSet's and then process that vector in the right order in
79  // generate(). But, the ordering difference is unlikely to change since it would harm the
80  // performance of the liveness analysis.
81  //
82  // Currently, we do (1) for B3 stackmaps.
83  virtual void reportUsedRegisters(Inst&, const RegisterSet&) = 0;
