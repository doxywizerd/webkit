217 // InlineWatchpointSet is a low-overhead, non-copyable watchpoint set in which
218 // it is not possible to quickly query whether it is being watched in a single
219 // branch. There is a fairly simple tradeoff between WatchpointSet and
220 // InlineWatchpointSet:
221 //
222 // Do you have to emit JIT code that rapidly tests whether the watchpoint set
223 // is being watched? If so, use WatchpointSet.
224 //
225 // Do you need multiple parties to have pointers to the same WatchpointSet?
226 // If so, use WatchpointSet.
227 //
228 // Do you have to allocate a lot of watchpoint sets? If so, use
229 // InlineWatchpointSet unless you answered "yes" to the previous questions.
230 //
231 // InlineWatchpointSet will use just one pointer-width word of memory unless
232 // you actually add watchpoints to it, in which case it internally inflates
233 // to a pointer to a WatchpointSet, and transfers its state to the
234 // WatchpointSet.

343  // Note that for any watchpoint that is visible from the DFG, it would be incorrect to write code like:
344  //
345  // if (w.isBeingWatched())
346  // w.fireAll()
347  //
348  // Concurrently to this, the DFG could do:
349  //
350  // if (w.isStillValid())
351  // perform optimizations;
352  // if (!w.isStillValid())
353  // retry compilation;
354  //
355  // Note that the DFG algorithm is widespread, and sound, because fireAll() and invalidate() will leave
356  // the watchpoint in a !isStillValid() state. Hence, if fireAll() or invalidate() interleaved between
357  // the first isStillValid() check and the second one, then it would simply cause the DFG to retry
358  // compilation later.
359  //
360  // But, if you change some piece of state that the DFG might optimize for, but invalidate the
361  // watchpoint by doing:
362  //
363  // if (w.isBeingWatched())
364  // w.fireAll()
365  //
366  // then the DFG would never know that you invalidated state between the two checks.
367  //
368  // There are two ways to work around this:
369  //
370  // - Call fireAll() without a isBeingWatched() check. Then, the DFG will know that the watchpoint has
371  // been invalidated when it does its second check.
372  //
373  // - Do not expose the watchpoint set to the DFG directly, and have your own way of validating whether
374  // the assumptions that the DFG thread used are still valid when the DFG code is installed.
375  bool isBeingWatched() const
376  {
377  if (isFat())
378  return fat()->isBeingWatched();
379  return false;
380  }
