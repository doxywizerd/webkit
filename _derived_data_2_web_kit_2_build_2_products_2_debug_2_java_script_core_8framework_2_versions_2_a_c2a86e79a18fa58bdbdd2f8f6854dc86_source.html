34 // A table of inferred types for some structure. This is a JSCell because that simplifies the Structure
35 // destructor and makes lifetime easier to manage. For example, since it's a cell, we know that this thing
36 // cannot be deleted while the DFG is running. This is separate from PropertyTable because most properties
37 // will not have an InferredType, since most properties are in dictionaries (if you think of "a property"
38 // as being "a property in some Structure"). Also, this will happily conflate the types of properties from

84  // Bottom: absence from table.
85  // Top: null value in table, !value->isRelevant(), or absence from table.
86  //
87  // We know how to determine if absence from the table is bottom or top depending on whether the
88  // property is present in the owning structure. Hence, depending on the structure, absence may have
89  // different meanings. The way that this class determines if the owning structure knows of the property
90  // is that we differentiate between actions: replace or transition. If we're adding a new property
91  // (transition), then absence means bottom. If we're storing to an existing property (replace), then
92  // absence means top. To make this work, we have to watch out for the case where two structures, S1 and
93  // S2, share the same InferredTypeTable and neither of them initially know about property P. S1 may
94  // want to add P without a type, while S2 may want to add P with a type. If S1 added P and used absence
95  // to indicate Top, then S2 would use that absence to mean Bottom and would end up creating a non-Top
96  // entry in the table. Then S1 would forget that it wanted Top, and would use S2's type. Clearly,
97  // that's bad. We avoid such confusion by ensuring that a transition always adds an entry. Hence,
98  // absence-means-bottom only comes into play for properties added before the InferredTypeTable was
99  // created.
100  typedef HashMap<RefPtr<UniquedStringImpl>, WriteBarrier<InferredType>, IdentifierRepHash> TableType;
101 
102  TableType m_table;
103 
104  // We only grab this lock when we're doing modifications on the main thread, or reads on the compiler
105  // thread. The compiler thread is not allowed to do modifications.
106  ConcurrentJSLock m_lock;
